Demo: Data shapes

1. Data comes in many forms

Structured datasets do not always come arranged in rows and columns, the format R is best at reading and processing. But most forms of structured data can be "reshaped" into rows and columns. In this tutorial, you will learn more about the non-rectangular data format known as JSON, how to reshape JSON data, and how to read and write data.

First, we will load some of the packages we need for this tutorial: 

#  run these first if you need to install the packages
#  install.packages("tidyjson")
#  install.packages("haven")
#  install.packages("tidyverse") # this one is bigger

  library(tidyjson) # for working with JSON data
  library(haven)    # for working with proprietary data formats
                                 # (like Excel, SPSS, or Stata)
  library(tidyverse) # a swiss army knife for data analysis

Packages usually include example datasets. tidyjson has JSON data, one of which is called "worldbank" (world bank funded projects).

First, we should take a look at the str-ucture of the data. 

# help(worldbank) # sometimes packages come with their own tutorials
str(worldbank)

Computer science people would call this a "500 length character vector." If you know what you're looking at, you might notice right away that the characters are probably in JSON format.

We can think of it more generally as an object consisting of 500 different units. Each unit happens to be encoded as collections of letters and numbers (characters). These collections happen to be structured in the JSON standard format.

What is "standard JSON format"? Let's look at the first unit of the object: 

worldbank[1]

Can you figure out how to look at the 250th unit? The 131st?

Not very nice to look at. A "human readable" display of JSON data looks more the following code chunk. Note that this code chunk is NOT WRITTEN IN R CODE. It is just here to help you understand how JSON is normally displayed. Do not attempt to copy/paste this into your R console or script — it will cause problems. 

{\"_id\":{\"$oid\":\"52b213b38594d8a2be17c780\"},
    \"boardapprovaldate\":\"2013-11-12T00:00:00Z\",
    \"closingdate\":\"2018-07-07T00:00:00Z\",
    \"countryshortname\":\"Ethiopia\",
    \"majorsector_percent\":[{\"Name\":\"Education\",\"Percent\":46},
                             {\"Name\":\"Education\",\"Percent\":26},
                             {\"Name\":\"Public Administration, Law, and Justice\",\"Percent\":16},
                             {\"Name\":\"Education\",\"Percent\":12}],
    \"project_name\":\"Ethiopia General Education Quality Improvement Project II\",
    \"regionname\":\"Africa\",
    \"totalamt\":130000000}

This entry has eight separate labels, each for different types of data. majorsector_percent is a label for another set of labels that name each sector's percentage of world bank funds. This is called nesting. JSON is great for nested data.

JSON is required knowledge for data analysts. But we won't discuss it beyond this demo. Have more questions? That's what Coffee and Code sessions are for.

2. From JSON to table

Usually we think of data in the form of tables, or (if you prefer) in "spreadsheets," a concept from accounting. What would the JSON entry for Ethiopia look like as a table? This is where the jsontidy package comes in handy! 

wb_table_ethiopia <- spread_all(worldbank[1]) # notice the <-
wb_df_ethiopia <- as.data.frame(wb_table_ethiopia)
                                     ## <- means define everything to the right under
                                     ## the object name to the left.
wb_df_ethiopia # invoking the object name returns the contents of the object

I assure you, this is a table. The nested variable is the reason it doesn't look very tidy. So let's just ignore columns with nested data for now. 

wb_df_ethiopia[2:7] # now we are "subsetting" columns 2 through 7 of the table

Now we are beginning to see how JSON translates into "tabular" form. Let's do the same thing but for the first 10 countries in the worldbank data. 

wb_table <- spread_all(worldbank[1:10])
wb_df <- as.data.frame(wb_table)
wb_df[2:7]

3. Export and import data in R

Export

We can export this table as a .csv file using the built-in function write.csv(): 

write.csv(wb_df[2:7], "worldbank_selection.csv")

Running the block above exports columns 2 through 7 to a file named worldbank_selection.csv. The file will be in your default working directory. Figure out what that is by writing the following: 

getwd()

Navigate to the directory given by the getwd() function in your file manager, then open up the worldbank_selection.csv file in your plain text editor. As will be obvious when you see it, .csv stands for comma separated value. Once you have seen the raw data in text form, open up the file in a spreadsheet application such as Numbers (on a Mac), Excel, or Google Sheets.

project_name is the only column that uses an underscore. Delete the underscore so the column name reads projectname. Then save the file as worldbank_selection_mod.csv in a folder called "data" within your DATA 306 folder (make one if you don't have one yet).

File paths and the working directory

We can read the data back into R by using the built-in read.csv() function. But first, we need to tell R where to look for the file with the setwd() function. This function requires a file path as an argument.

A file path is a nested list of folders leading to a given file. An example was given above in the output to getwd(). You see file paths everyday in the form of web addresses. The webpage for the Data Analytics program is https://qcsociology.commons.gc.cuny.edu/data-analytics-applied-social-research/. This just means "data-analytics-applied-social-research" is a sub-directory (folder) within the parent directory "qcsociology.commons.gc.cuny.edu".

Usually, your "Desktop" directory is within what is called the "home" directory. The home directory is important, so programmers abbreviate this path to "~", the tilde key. So to write a file path to your desktop, on a Mac you can use something like "~/Desktop/". If you created a DATA 306 folder on your desktop, then the path to this directory should be something like "~/Desktop/DATA306/". In Windows, the tilde key takes you all the way to the desktop, so you can just write "~/DATA306".

So, we can tell R to consider a folder called DATA306 as its "working directory." The working directory is where R looks for any imports (like data) and where it puts any exports (like data, plots, workspaces, or other outputs).

We can set the working directory like so: 

setwd("~/Desktop/DATA306")

Importing data

Assuming everything is configured as described above, importing the data should be straightforward with the read.csv() function.

Note that we use a file path from the working directory to tell R that the csv file is in the "data" folder. We add this because we want to segregate our outputs (modified data, plots, etc.) from our inputs (the original data). 

mydata <- read.csv("data/worldbank_selection_mod.csv") # read the data
str(mydata) # look at the data
